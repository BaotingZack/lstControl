# 方案2：统一 PD 控制循环架构

## 一、背景与目标

### 问题

方案1 中 `run_simulation()` 通过 `if/else` 分支同时处理仿真和 PLC 两种模式，导致：

1. **PD 流程不一致**：仿真模式走 100Hz 固定步长 + plant model，PLC 模式混合了定位数据和仿真 fallback
2. **Plant model 混入 PLC 模式**：无定位数据时自动回退到仿真对象模型，PLC 模式下被控对象是真实物理系统，不应运行仿真模型
3. **速度语义混乱**：`crane.x.velocity` 在有定位时 = `vx_cmd`，无定位时 = 伺服模型滞后速度
4. **前端竞态**：Control 标签页速度被 `renderStatus()` 和 `pollPlcStatus()` 两个函数争抢更新
5. **交互模式不匹配**：PLC 模式应在 UI 设置 Target 后才开始，而非命令行预设

### 目标

1. 两种模式通过 `--plc-ip` 区分，PD 算法完全共用
2. PLC 模式位置通过 `/localization_pose`（10Hz）获取，不做仿真 fallback
3. PLC 模式：开 UI → 实时显示位置 → 设 Target → Apply Target → 开始控制
4. 两种模式使用统一控制循环

### 接口定义

```python
class PositionSource:
    def get_position(self) -> dict | None:
        """返回 {'x','y','z', 'vx','vy','vz', 'dt','t','stamp'} 或 None(超时)"""

class Actuator:
    def apply(self, vx, vy, vz, dt) -> None:
        """执行速度指令"""
    def update_state(self, state, position) -> None:
        """从位置数据更新 CraneState"""
    def emergency_stop(self) -> None:
        """安全停止"""

class ControlHooks:
    def on_step(self, step_data) -> None: ...
    def on_arrival(self, axis, t) -> None: ...
    def should_stop(self) -> bool: ...
```

## 三、控制循环流程

```
run_pd_control(source, actuator, config, target, hooks):

  while not all_locked:
    │
    ├─ STEP 1: pos = source.get_position()
    │    模拟: 100Hz 非阻塞, 始终返回
    │    PLC:  10Hz 阻塞等待新数据, 2s 超时→安全停止
    │
    ├─ STEP 2-3: PD 计算 (两种模式完全共用)
    │    raw_v = Δpos/dt
    │    filtered_v = lowpass(raw_v, τ)
    │    v_cmd = Kp*error - Kd*filtered_v
    │    限幅 ±v_max
    │
    ├─ STEP 4: actuator.apply(vx, vy, vz, dt)
    │    模拟: plant.update_axis() → 一阶伺服 + 扰动
    │    PLC:  BigCarCtrl/SmallcarCtrl/liftctrl → Snap7
    │
    ├─ STEP 4b: actuator.update_state(crane, pos)
    │    模拟: 已由 plant.update_axis 更新
    │    PLC:  位置←定位, 速度←Odometry原生速度
    │
    ├─ STEP 5: 到达检测 (共用)
    │    |pos-target| < tol 且 |vel| < tol → locked
    │
    ├─ STEP 6: hooks.on_step(step_data)
    │    PLC 模式: 入队 → SSE → 浏览器实时渲染
    │    模拟模式: 仅记录 history
    │
    └─ 检查 hooks.should_stop() / 超时
```

## 四、10Hz vs 100Hz 的 PD 适配

| 参数 | 仿真模式 (100Hz) | PLC 模式 (10Hz) |
|------|-----------------|-----------------|
| dt | 固定 0.01s | 实际间隔 ≈ 0.1s |
| `alpha = 1-e^(-dt/τ)` | τ=0.25s → α≈0.039 | τ=0.50s → α≈0.18 |
| raw_velocity 噪声 | 高 (Δpos/0.01) | 低 (Δpos/0.1) |
| 速度滤波响应 | 慢 (α小, ~85ms 达63%) | 中 (α大, ~100ms 达63%) |

PLC 模式使用更大的 `velocity_filter_tau_plc = 0.50s`，以补偿大 dt 带来的 alpha 增大，保持相似的噪声抑制特性。

起重机的机械时间常数（秒级）远大于 0.1s 采样间隔，10Hz 控制频率对于大惯性系统足够。Nyquist 频率 5Hz 也远超起重机摆振频率（< 0.5Hz）。

## 六、PLC 模式交互流程

```
1. python3 main.py --live --plc-ip 192.168.1.100

2. 后端: 连接 PLC → 启动心跳 → 启动 ROS → 等待首次定位 → 开 HTTP 服务

3. 浏览器打开 http://127.0.0.1:8000
   - Canvas 显示 "Waiting for localization data..."
   - Localization 标签页: SSE 实时显示 /localization_pose 数据
   - Control 标签页: PLC 连接状态 + 心跳
   - Apply Target 按钮可用

4. 用户在 Target Command 区输入 X/Y/Z → 点击 [Apply Target]

5. 前端 POST /api/start-control {"target_x":8,"target_y":6,"target_z":1.5}

6. 后端: 获取当前定位 → 创建 LiveControlHooks → 启动控制线程
   - 控制线程: run_pd_control(ros_source, plc_actuator, ...)
   - 每步通过 hooks.on_step() 入队

7. 前端: 订阅 /control/stream SSE
   - 每收到 step → 更新 Readout 位置/速度, Canvas 实时渲染
   - 收到 arrival → 提示 "X arrived at 12.3s"
   - 收到 done → "Control complete"
   - 收到 error → 显示错误信息

8. 控制完成后可重新设置 Target 再次运行

9. 随时可点击 STOP ALL 发送零速度 (匹配 demo.cpp 模式)
```

## 七、与方案1 的关键区别

| 维度 | 方案1 | 方案2 |
|------|-------|-------|
| 控制循环 | 一份代码，if/else 分支 | 统一循环 + 策略注入 |
| 仿真 fallback | PLC 模式无定位时自动跑 plant model | 无 fallback，超时→安全停止 |
| PD 核心代码 | 一份，但混合模式切换 | 完全共用，模式差异在 source/actuator |
| 速度来源 | 有定位=v_cmd，无定位=servo | 始终从 source 获取，PLC 用 Odometry |
| PLC 模式交互 | 命令行 target → 自动运行 → 回放 | UI 设置 → Apply Target → 实时控制 |
| 前端渲染 | 仿真完成后回放 | SSE 实时推送 + Canvas 实时跟踪 |
| ctrlVx 竞态 | renderStatus 和 pollPlcStatus 冲突 | pollPlcStatus 独占更新 |
| 可测试性 | 混合逻辑，难以单独测试 | source/actuator 可独立 mock |
